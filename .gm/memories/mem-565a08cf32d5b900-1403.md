gm-method (witnessed): fixing a repeat/parrot guard by adding a content-splicing fallback template ('I got that: "X". Tell me about Y and Z') can itself become the next visible bug -- a FIXED compound follow-up question presumes what is and isn't already known, so it will eventually re-ask for something the user just said, in a system that deliberately does no deterministic field extraction (so the fallback structurally CANNOT know what's already covered). The durable shape for a content-echoing fallback in a no-extraction system: acknowledge the input, then ask ONE OPEN, NON-PRESUMPTIVE follow-up ('tell me more about what's going on') rather than a fixed multi-part question naming specific fields -- open questions can never contradict information already given, fixed compound questions eventually will. Also witnessed: live-verifying a fix by driving real handleInbound calls against a REAL production user's actual channel/author id (to prove the fix against the exact reported bug case) does prove the fix, but permanently mixes synthetic test messages into that user's real conversation history -- the fix is a non-destructive append-only 'dev note' event on the affected record, never an edit/delete of real history; the durable practice is to reserve a witness id-space that structurally cannot collide with production ids before running any live-witness test that touches shared state.
