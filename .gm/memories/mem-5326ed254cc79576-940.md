gm-method (witnessed): for a large multi-repo re-architecture, run a DESIGN-FIRST fan-out workflow (map each repo -> design each layer -> adversarial cross-repo coherence verifier) BEFORE writing any code. The verifier defaulting-to-refute caught six fabricated-API violations in the first design draft -- e.g. it assumed a SQL query-engine that did not exist when the real store was a document store with a different builder, and assumed a config knob that required an unpublished upstream code change. Applying that draft directly would have failed immediately on nonexistent files. The cost of the design pass is repaid many times over by not editing against hallucinated surfaces. Also: when a dependency publishes via CI on push and the consumer pins a version range, order the work dependency-first (upstream code -> publish boundary -> consumer) and add a feature-detect shim in the consumer so it stays green across the publish lag.
