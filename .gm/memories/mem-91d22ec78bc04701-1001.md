gm-method (witnessed): an intermittently-failing integration test (green some runs, N-failing others, SAME code) is almost never the test -- it is a CONCURRENCY bug the change exposed. Root-cause by reading the durable audit trail, not by re-running: here the case/event log showed "agent turn error: SQLITE_BUSY: database is locked" -- an agent tool reading an embedded-sqlite store concurrently with the same turn's writes threw out of the turn and produced the degraded fallback instead of the real answer. Fix at the store boundary: wrap the mutating/reading methods in a bounded retry on the BUSY/locked error class with small linear backoff (only that class retries; other errors propagate). Witness the fix by running the suite MULTIPLE consecutive times (3/3 green), not once -- a single green run does not prove a concurrency fix. Build a tiny repro harness (spin up the real store + stub, drive the exact 2-message sequence, dump the case event kinds) to SEE the error class before guessing.
