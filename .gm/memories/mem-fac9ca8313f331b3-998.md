gm-method (witnessed): when a user asks to maximize effort at a specific in-conversation moment ('strike while the iron is hot' before a one-shot window closes), the audit target is the prompt's OWN internal consistency first -- grep for near-duplicate instructions covering the same behavior at different points in a large system prompt (they dilute rather than reinforce a small model's compliance), check whether the instruction explicitly names the SAME canonical field/entity list a downstream guardrail or aggregate already uses (unnamed vague language like 'an important fact' makes the model guess what downstream code already knows precisely), and check whether the instruction is temporally anchored to the actual state-transition moment that matters (e.g. explicitly 'before you declare complete', not just 'when they seem to be leaving') -- merging two weak echoes into one strong, precisely-targeted, correctly-timed instruction is usually the real fix, not adding a third instruction.
