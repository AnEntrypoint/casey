gm-method (witnessed): a long-lived conversation/session that REUSES one durable entity (a case/ticket/thread) across many user intents creates a recurring dead-end class: when the entity is in a TERMINAL/COMPLETE state and the user starts something genuinely NEW, the handler treats the new input as activity on the finished entity and emits the terminal/closed message ('we are done'). The fix is not another guard on the closing message -- it is to BRANCH a fresh entity (or re-open) when the input is a genuine new start, bound by a STRUCTURAL continuation guard (a present-AND-different key, not a fuzzy heuristic) so a real continuation is never mis-branched and an incomplete entity never branches. Pair it with defence-in-depth at the close-emit site (a predicate that only emits the terminal message on a genuine completion arc -- a completing answer this turn OR no new differing key). Watch two adversary traps the verifier must catch: (1) N>2 siblings sharing the reuse key need a wide-paged newest-wins so rebinding lands on the freshest, not a stale terminal one; (2) the delta predicate must be tight -- an entity-type word alone in a wrap-up ('thanks, the cattle are fine') must NOT block the terminal message, only a DIFFERING key. Also: a list/itinerary line should render the entity's CONTENT (subject), never a verbose per-entity STATUS SENTENCE meant for a 1:1 reply -- the sentence reads as a stray holding reply in a list.
