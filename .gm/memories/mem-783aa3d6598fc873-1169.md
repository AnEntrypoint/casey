gm-method (witnessed): a row-access / permission-scoped read can silently return ~0 rows for a caller who is not in the scoped set, and the calling code then renders a CONFIDENT WRONG answer ('0 open', 'nothing overdue') that LOOKS safe -- a reassuring-when-wrong false-safe far more dangerous than an error. When an aggregate/count surface reads through a scoped query, verify the CALLER is actually in the scope (here the chat author is the reporter, but row_access scoped to assignee). For a surface that only emits PII-free scalars, the correct fix is to read UNSCOPED and rely on the projection for safety, keeping the scoped read only where per-caller filtering is the actual intent. Also: when adding a fixed-intent branch that short-circuits before the main turn (a ref/status lookup), gate it on a GENUINE signal (real report-content words), not the classifier's catch-all default -- a too-broad guard ('any report-kind') blocks the very inputs the branch exists for. And a per-call typo/precedence fix needs the FULL existing matrix re-run: a new high-precedence trigger ('unanswered'->overdue) can steal a different intent ('how many ... unanswered'->count).
