gm-method (witnessed): a fill-if-empty merge into a durable entity creates a silent dead-end when the entity is already COMPLETE -- a user's genuinely-new input no-ops (every field already set), the system detects 'nothing changed' as 'done', and recites the terminal message while the new work is LOST and the user is even falsely told it was recorded. The branch-a-fresh-entity fix must not require the new input to CONFLICT with the old (a differing key) -- a user re-doing the same kind of thing (same place/species) carries no conflict yet is still a new instance. The correct signal is: the entity is complete AND the input carries real content of the entity's type AND that content NO-OP'd (nothing newly captured). On a COMPLETE entity there is nothing left to add, so any new typed content is a new instance -> branch. Guard it on complete-only so a continuation on an INCOMPLETE entity (which legitimately adds a missing field) never branches. Witness the live entity's actual stored state (not your assumption) to confirm it is complete -- the bug was invisible until reading the real record showed all fields already filled. And a 'noted X' ack must be tied to a REAL fill (the merge's filled-set), never fired on a no-op, or the user is told their input landed when it did not.
