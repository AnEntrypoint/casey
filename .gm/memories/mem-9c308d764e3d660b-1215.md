## Resolved mutable: acptoapi-chat-drops-tool-choice

Confirmed via direct read of acptoapi/bin/acptoapi.js (the actual HTTP server binary casey currently talks to over :4800): it contains ZERO references to tool_choice anywhere. The HTTP path casey uses TODAY already silently drops tool_choice before reaching any provider -- toParams()/formats/openai.js never reads req.tool_choice, and the only place that field is read (lib/transformers.js 'tooluse' transformer) is an opt-in HTTP-layer transformer never wired into the default server request path. So switching to in-process chat() is NOT a regression -- tool_choice enforcement was already a no-op over the OLD HTTP path too. This likely explains part of why chatjimmy/llama3.1-8B has been observed never calling case_report all session: casey/freddie's tool_choice:'required' plumbing (AGENTS.md's documented layering mandate) has never actually reached the provider through acptoapi at all, HTTP or in-process. Not a regression introduced by this change; a pre-existing acptoapi gap, out of casey/freddie's repo boundary to fix (acptoapi is a separate third-party npm package) -- documented as an operational finding, not blocking the in-process migration.
