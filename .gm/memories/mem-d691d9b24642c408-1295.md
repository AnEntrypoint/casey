gm-method (witnessed): a finite-string blocklist guard (isStockAck-style: compare output against a hardcoded list of known-bad phrasings) is structurally incomplete the moment the guarded text changes -- fixing one instance of a canned-reply bug by adding a NEW canned string, without also adding that new string to every existing detection guard, silently reopens the exact same bug class under new text. Symptom: a live user reported the same identical-reply bug persisting AFTER a fix landed; the fix was real (root cause A was fixed) but introduced NEW text that a SEPARATE guard (meant to catch exactly this shape of problem) was blind to, because the guard's list was never updated. The durable fix is not another blocklist entry -- replace the finite-list comparison with a STRUCTURAL check against the actual prior state (e.g. compare a new reply against the entity's own last recorded output, not against a maintained list of forbidden strings) so the guard remains correct automatically as the guarded text evolves. When debugging a 'still broken after the fix' report, always re-verify end-to-end against the LIVE running process with a real witness (not a synthetic unit-level check) before declaring victory a second time -- the first fix's own verification missed exactly this gap.
