gm-method: when a user re-reports 'service not responding' for a defect you believe is fixed, the FIRST witnesses are (a) is the channel/socket actually connected vs the model/backend offline -- a health endpoint that re-probes per request tells the live truth, while a backend captured once at boot lies; and (b) does the running process predate your fix commit (compare process StartTime to the fix commit time) -- a correct committed fix is inert until restart when hot-reload keys on file mtimes, which git commit/checkout does not touch. A boot-time one-shot probe whose result is closed over for the process life is a latch: the resilient shape is a lazy re-resolving wrapper (single in-flight probe, debounced, throws-while-degraded so the existing fallback fires) with one live status() shared by the health row, so recovery needs no restart. Never add a probe round-trip to every hot-path call -- debounce and share the in-flight promise.
