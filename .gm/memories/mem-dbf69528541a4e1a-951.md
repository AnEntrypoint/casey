gm-method (witnessed): when adding an optimistic-concurrency guard to a data store, do not reuse an existing coarse-precision timestamp column (e.g. a second-precision now()) as the version discriminant -- live-testing a two-writer race found the guard silently ineffective because both writes landed within the same second, producing an identical stamped value that made a genuinely stale write's conditional check pass instead of correctly conflicting. Use a dedicated strictly-monotonic integer counter column instead, incremented on every write regardless of whether the guard is invoked for that particular caller -- this has no precision ceiling and is unaffected by clock/timestamp granularity. Always live-test the guard with a genuinely injected race (not just a sequential read-then-write with an artificial delay) before trusting it: sequential tests can pass even when the underlying precision gap would let a real concurrent race through.
