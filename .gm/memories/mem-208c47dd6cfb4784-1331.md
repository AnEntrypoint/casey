gm-method (witnessed): when a live LLM agent REPEATS a question the user already answered, or fabricates context on an empty turn (thanks for a report on a bare greeting), and the reply string is NOT in the app source, it is the MODEL's output steered by the system prompt -- fix the PROMPT, not deterministic code. Two prompt levers: (1) surface the ALREADY-CAPTURED state explicitly ('You ALREADY know: <fields>') AND name the single NEXT missing thing to ask ('ask about THIS next -- <fact>') AND forbid repetition ('do NOT ask the same question you asked last turn') -- a passive 'never re-ask what is in the record' is too weak; the model needs the positive next-action. (2) branch the opener on empty-vs-nonempty state so it does not assert a report that was never made. Make it testable without the live model by EXPORTING the prompt builder and asserting the built string contains the known-fields + next-ask + no-repeat instructions for a given case state (a prompt-content assertion is the reachable proxy for model steering). Also: the deterministic extraction FLOOR that feeds 'report so far' must actually capture the lay terms users type (e.g. 'pussy/weeping/runny eyes' for eye discharge) or the prompt shows nothing-known and the model re-asks -- broaden the floor vocabulary when a real message drops a real field.
