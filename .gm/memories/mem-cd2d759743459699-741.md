## Resolved mutable: mut-reload-mechanism

exec_js-142 (Node v24.15.0): same import specifier shares the module singleton (cached, state persists); a cache-busted specifier '?v=x' FORKS the module into a fresh instance with reset state (cachebust_forks_module=true, cachebust_resets_state=true). Therefore in-process ESM cache-busting cannot hot-swap a changed module without forking its module graph -- a cache-busted re-import of thatcher/freddie would fork their primed store/db singletons into a second invisible handle (the exact mem-44264 hazard). RESOLUTION: reload MUST be parent-supervisor + child-worker respawn; the sqlite FILE (reopened by the fresh worker via the same cwd/home) is the durable boundary, not an in-memory handle.
