gm-method (witnessed): when one surface (a chat) is asked to reach the breadth of another (a GUI/dashboard), the highest-leverage move is to find that the rich surface is already built on PURE functions over a shared store -- aggregators like build<Thing>(items, eventsByItem) -- and CALL THE SAME functions from the lean surface, rather than reimplementing. The verification that matters then is not 'does it compute' but the cross-cutting INVARIANT the new surface must also honor (here: a field worker must never see PII, so the aggregate answer renders counts/tokens only and never iterates a structure whose members carry contact free text). A default-refute verifier pass catches the real traps that breadth-by-reuse introduces: wrong field name on the aggregator result, an unbounded fan-out (cap the event hydration), a self-gating trigger word (an 'overdue' regex gated on 'late' fires on 'my order is late'), and a too-broad token ('total' in a count-head matches '10 total animals', a report). Always run the new classifier against the EXISTING scenario corpus, not just the new matrix -- the two bugs here only surfaced against the old report scenarios.
