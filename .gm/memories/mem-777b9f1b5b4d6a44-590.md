gm-method (witnessed): `npm install <pkg>@latest` rewrites a package.json dependency entry FROM a bare "latest" tag TO a resolved semver range (e.g. "^1.0.37") -- this silently reverses a repo's explicit policy of "always resolve latest at install time, never pin" if that policy exists. After running npm install to pick up a freshly-published sibling-repo fix, always diff package.json afterward and revert the dependency-spec line back to its intended form (a bare tag/range) before committing -- the goal was picking up new code via a reinstall, not changing the version-pinning policy.
