gm-method (witnessed): when adding a new agent-facing tool param constrained by a JSON-schema enum, do not assume the enum is enforced at the storage layer -- live-test writing an out-of-enum value through the actual handler and reading it back. Found instance: a config-driven ORM's declared enum type was NOT enforced server-side on write; only a UI-layer check (a separate hand-maintained validation in the dashboard route) caught bad values, and a newly-added agent tool exposing the same enum field had no equivalent guard until a live round-trip test caught the silent corruption. The fix pattern: validate against the SAME allowed-values set the existing UI-layer check already uses, before every write, at every entry point that accepts the field -- a schema-level enum is a hint to a well-behaved caller, never a runtime guarantee.
