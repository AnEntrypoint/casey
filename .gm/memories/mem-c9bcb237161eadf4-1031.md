casey/thatcher fact (witnessed 2026-07-12): thatcher's list() DOES honor sort server-side (busybase-store.js:179-192 sorts on options.sort OR spec.list.defaultSort) -- the old casey comments claiming 'thatcher ignores orderBy/order' were stale and false; casey's listCases already passes sort: and thatcher.config.yml declares list.defaultSort. BUT casey deliberately keeps JS sort in the EVENT paths (listEvents/listEventsPage via sortByCreatedStable) because event created_at is coarse unix-SECONDS: a whole turn's events share one second and thatcher's comparator has no insertion-order tiebreak, so pushing the event sort down would scramble same-second replay order. Contacts/refs have no such tiebreak need so their sort pushes down cleanly. Also: findOpenCase must use status:{$in:openStages} (allowlist) NOT status:{$ne:'closed'} -- busybase bypasses its auto-DELETED soft-delete filter when status is IN the where clause, so a $ne denylist leaks status='deleted' rows while the open-stage allowlist excludes them for free.
