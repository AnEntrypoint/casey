gm-method (witnessed): a user directive to remove hand-curated hardcoded lists in favor of trusting the model's own judgment is a genuine RESHAPING discovery when it reverses a prior session's own additions -- audit every hardcoded list/table in the codebase against a two-question test: (1) is this a deterministic safety/legal-necessary control that must work even with the model unavailable (an opt-out/consent keyword layer)? (2) is this storage-integrity validation that guards data AFTER the model has already exercised judgment via a tool call (an enum check preventing corruption), not a stand-in for interpretation? Only tables that fail BOTH tests are the genuine hand-curation to remove -- a table that looks similar (e.g. also a keyword list) but is actually the deterministic safety layer must be explicitly reasoned about and kept, or a well-intentioned 'stop hand-curating' sweep would strip a load-bearing control the user did not intend to lose. When removing, don't just delete -- widen the corresponding prompt/tool-description language to explicitly grant and trust the model's expanded mandate, or the removed determinism leaves an unfilled gap rather than the intended expanded interpretation.
