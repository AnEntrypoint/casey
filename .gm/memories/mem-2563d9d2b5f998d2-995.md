gm-method (witnessed): when live-witnessing a rate-limit/burst/timing-window scenario against a real slow backend, the WITNESS SCRIPT's own send pattern matters as much as the code under test -- awaiting each call serially before sending the next silently converts a 'rapid burst' test into a 'spaced-out' test if each call takes longer than the rate window (15 messages awaited serially at ~35s/call against a 60s window never accumulates more than 1-2 in-window hits, producing a false FAIL on correct code). The fix is not to loosen the code's threshold to match slow test timing -- it is to fire the witness calls CONCURRENTLY (Promise.all, not a for-await loop) so the witness actually exercises the timing behavior it claims to test, independent of downstream latency. Before concluding a rate/concurrency/timing guard is broken from a live witness, check whether the witness's own dispatch pattern (serial vs concurrent) matches the real-world trigger condition it is trying to reproduce.
