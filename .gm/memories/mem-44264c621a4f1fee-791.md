gm method on a target with an embedded store dependency: when a library primes a process-global handle argless at init (e.g. a DB opened from cwd), you cannot relocate it by importing the library's internal getter yourself -- that forks the module graph into a second instance with its own handle, invisible to the real one. The only safe relocation is the dimension the argless prime reads (here, process cwd). Witness which file the handle actually opens before trusting a config/env override; an override that the argless prime never consults silently does nothing. A destructive test that rmSync's a fixed dir shared with a live process is a critical defect: isolate the test by giving it its own cwd/dir and anchor any sibling-dep imports to a captured repo-root, not the relocated cwd.
