## Resolved mutable: mut-timestamp-corrupt-masks-urgent

case-health.js classifyCaseHealth() (line ~73-90): const touched = lastTouch(caseRow); if (!Number.isFinite(touched)) { out.push({breach:'timestamp_corrupt', ...}); return out } -- this is an EARLY RETURN before any of the subsequent checks (stale/stuck/unanswered_handoff/unsent_draft/abandoned_intake/incomplete_critical) ever run, for ANY case whose lastTouch is non-finite regardless of its tags. CONFIRMED REAL as a code path, but LOW real-world impact: unanswered_handoff/unsent_draft depend on `idle = now - touched`, which is undefined/NaN when touched is non-finite anyway, so those checks could not fire correctly even without the early return (idle >= threshold would be false for NaN). The early return does not suppress a check that would otherwise have fired correctly -- it fires the SAME single generic breach a broken idle computation would produce regardless. Net: a timestamp-corrupt case surfaces distinctly (as intended) rather than silently, and attn.js's caseHints/attnScore never reads case-health's breach list directly for tag-based scoring (needs-human/draft-pending tags on the case row itself, set independently of classifyCaseHealth, still score normally in attnScore) -- so a needs-human-tagged case with a corrupt timestamp STILL scores +100 in attnScore and surfaces top of the inbox; only the classifyCaseHealth-driven health:* tags (stale/stuck/unanswered_handoff/etc, which the periodic sweep would otherwise add) are what's skipped for that case until timestamps are fixed. This is a real but narrow gap: NOT a masking of an already-flagged urgent case, but a missed opportunity to flag NEW breaches on a case with corrupted timestamps.
