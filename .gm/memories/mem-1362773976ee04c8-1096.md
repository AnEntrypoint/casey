gm-method (witnessed 2026-07-12): the gm plugkit watcher (0.1.851/0.1.853) repeatedly died mid-verb during embed.wasm-loading on a LARGE memory store -- casey's .gm/memories/ had grown to 1185 .md files, and each boot re-syncs a batch (~10-13) through the 133MB BAAI/bge-small-en-v1.5 embed model, which combined with init can exceed the supervisor's hardcoded STATUS_STALE_MS=30000 heartbeat window (no env override), so the supervisor kills the worker mid-init -> restart loop where it never finishes. Recovery that WORKED: PowerShell-kill ALL plugkit procs (bun/node matching plugkit|wrapper|supervisor), rm the stale .status.json+.turn-summary.json, then bun x gm-plugkit@latest spool from the CORRECT repo root (running spool from a subdir creates a nested .gm/exec-spool/.gm junk dir + wrong-cwd watcher). A transition that triggers auto_recall+embed can take 10-23s; poll the .watcher.log for '<- verb=transition task=N' completion rather than the frozen status ts, and read .turn-summary.json for true phase after any verb_aborted (side effects often DID apply despite the abort message).
