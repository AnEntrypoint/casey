## Resolved mutable: mut-workload-stale-claims-snooze-interaction

workload.js buildWorkload() (line ~50-114): stale_claims is incremented via `const last = lastActivityMs(c, events); const waited = last != null ? now - last : null; if (waited != null && waited >= staleMs) k.stale_claims++` -- this path never reads c.tags, never calls snoozedUntil, and has no snooze-awareness of any kind, unlike attn.js's attnScore which explicitly exempts a snoozed case (unless needs-human) from being surfaced. CONFIRMED REAL: an operator who deliberately snoozes a case they cannot finish yet (the exact scenario attn.js's snooze comment describes: 'an operator has seen but cannot finish yet') still has that case counted as a stale_claims strike against them in the Team workload panel once staleMs elapses, identical to a genuinely dropped/forgotten case. This makes the worst-first Team ranking penalize deliberate, communicated triage decisions the same as neglect -- a false signal for team management, though not a wrong-behavior bug in the CONTACT-facing inbox (which does correctly hide snoozed cases from urgency).
