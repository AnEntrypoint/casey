gm-method (witnessed): a re-reported 'still broken' bug after a fix is pushed may be a STALE RUNNING PROCESS, not a code regression. Before re-diagnosing the code, witness two things: (1) run the suspect function against the REAL persisted data via exec_js -- if it now produces the correct output, the code is fixed; (2) compare the running process's start time to the fix commit's time -- a long-lived supervised worker that hot-reloads on source-mtime does NOT refresh on git push/pull, so a worker started before the commit serves stale code. The fix is a restart, not an edit. Also: persisted data poisoned by the OLD bug (e.g. a malformed field) survives the code fix and must be cleaned separately + guarded at the capture edge.
