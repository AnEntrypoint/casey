gm-method: a long-lived service process holds the code it booted with, NOT the latest commit. When a user re-reports a defect you believe you fixed, FIRST witness whether the live process predates your fix (check its CreationDate vs the fix commit time) before re-investigating the code -- the fix can be correct and committed yet inert until restart. Restarting a user-owned live process is an outward-facing action: confirm before killing it, then relaunch detached so it survives the session. Separately: stored recall/Q-A memos of a weak model's bad output (e.g. a parroted stock reply) actively TRAIN the model back toward that output via auto-recall; prune them on sight as a root cause, not just a symptom.
