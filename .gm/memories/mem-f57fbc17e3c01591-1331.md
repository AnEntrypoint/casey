gm-method (witnessed): when a user reports 'no improvement' after you committed + tested a fix, FIRST eliminate whether the live surface is even running your code before assuming the fix is wrong -- read the running process's start time vs the fix commit time, and witness which pid actually serves the user-facing surface. Here the live worker WAS fresh (forked after the commit), which proved the fix itself was incomplete (a new phrasing the broadened pattern still missed) rather than a deploy-lag -- two very different root causes that look identical from the user's seat. Second lesson: a deterministic fast-path that depends on an UNRELIABLE upstream (a weak LLM that rarely returns the forced tool call) cannot lean on that upstream for common cases -- the deterministic layer must cover the common inputs itself, and you make the upstream's contribution OBSERVABLE (log keys-added) so an always-empty contribution is the signal to broaden the deterministic layer, not to keep re-asking. Third: 'the running copy must auto-restart' is two mechanisms -- a file-watch reload (works on a local save/touch) AND a periodic pull (the missing piece: without a running auto-pull loop, a pushed commit never lands on an unattended host); ship both and witness the pull loop is actually a live process, not just a script that exists.
