gm-method debugging lesson (witnessed): when a real-services test of an aggregator/rollup returns a silent ZERO or empty-group where real input clearly exists (reply credit 0, notes never group, dwell mis-buckets), the prime suspect is a SERIALIZATION BOUNDARY -- a column persisted as a JSON STRING but read as an object via `obj.field` (a string has no such key, so the read is undefined and the count is silently 0, never an error). Process of elimination: the test failure named the exact assertion (replies_24h>=1), `codesearch`/Read of the store layer showed appendEvent stores `JSON.stringify(data)` while listEvents returns it unparsed, and a sibling test already parsed defensively -- three witnesses converging on the boundary. Fix at the lowest shared point (one exported parse helper, or parse once at the API boundary) so every consumer gets the object form, then collapse the duplicate inline parsers. One such bug is a tell-tale: sweep `obj.data?.X` reads across the codebase, they recur. The failing assertion is the cheapest locator; never guess-and-restart.
