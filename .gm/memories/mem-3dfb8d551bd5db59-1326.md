gm-method (witnessed): when a user says 'we still have to manually restart' despite a working hot-reload, the gap is usually that hot-reload only fires on a local file-MTIME change, but the user's real loop is push-from-elsewhere -> the running host never sees the new code. The fix is an AUTO-DEPLOY poll on the running host (fetch origin + fast-forward on an interval) that rewrites watched source so the supervisor reforks -- and it must be DEFAULT-ON, because an opt-in flag the user does not set is the same as absent. Making it default-on is SAFE for a dev checkout when the update is fetch + `merge --ff-only @{u}`: a dirty or divergent tree makes ff-only REFUSE and leaves the working tree untouched (witnessed: 'your local changes would be overwritten', edit survives), so a dev with uncommitted work just skips the pull. Prefer `git fetch origin` + `git merge --ff-only @{u}` over bare `git pull --ff-only`: the bare pull fails 'Cannot fast-forward to multiple branches' when FETCH_HEAD carries several refs (an all-branches fetch refspec) -- witnessed as per-interval failure spam.
