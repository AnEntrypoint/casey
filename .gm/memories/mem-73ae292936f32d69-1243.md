gm-method (witnessed cross-repo design verify): a multi-layer design doc can cite file:line witnesses that look precise but are FABRICATED -- always open the cited file before trusting it. Failure mode seen: a "core enabler" change targeted a SQL query-engine.js buildSpecQuery() with `col = ?` / positional params, but the real store was a document store (Supabase-style builder over a vector DB, Bun-required) with an equality-only applyWhere using .eq() chains -- no such SQL file existed. The capability (range/IN/OR operators) WAS reachable, but on a different file and different mechanism (builder .neq/.gt/.in/.or, not SQL). Lesson: verify (a) the file exists, (b) the mechanism matches (SQL vs document builder), (c) the cited line numbers are real, (d) hardcoded field names in a permission/row-scope check (e.g. record.assigned_to) match the consuming project's actual field (e.g. assignee) -- a config "scope: assigned" silently never matches if the column name differs and the check hardcodes the name. Also: a per-row enquiry/list tool is a NEW PII-leak surface even when the project has a strict aggregate-only no-identifier discipline elsewhere -- aggregate scrubbing does not cover per-row list outputs reaching an LLM context.
